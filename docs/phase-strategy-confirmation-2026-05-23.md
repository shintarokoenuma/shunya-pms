# Phase 戦略再確認（2026-05-23）

## 確認の背景

Phase 1A-6（外注先マスター）完了後、AI 機能の前倒し実装を検討。
具体的には以下の業務を AI で自動化したいという要望があった：

- メールを読み込んで品番ごとに内容を精査
- Meeting 音声を読み込んでタスク管理
- システム内での仕様書自動作成
- 工場・仕入先への発注書の自動作成
- 工場・仕入先からの請求書の自動入力
- 自動見積もり機能

## 設計書の確認結果

既存の設計書（`docs/20260516_03_仕様書_Part3_機能要件.md` 第7部「AI・自動化」）に
これらすべての AI 機能が既に網羅されていることを確認。設計不足はなし。

設計書での AI 機能の配置：
- Phase 1E： AI 機能コア（メール振り分け・議事録要約・翻訳）
- Phase 2： AI 読み取り機能（メール・PDF・Excel・紙）+ 仕様書 AI 生成

Prisma スキーマにも実装基盤が用意されている：
- `EmailMessage` モデル（AI 振り分け対応、関連エンティティ自動紐付け含む）
- `EmailAttachment` モデル
- `EmailClassification` enum（業務分類 13 種）
- `EmailStatus` enum（処理状態 10 種）

## 確定方針

### シナリオ採用

**シナリオ A 採用**： 設計通りに進める

```
Phase 1A: マスター実装（現在）
  ↓
Phase 1B: 見積もりエンジン（最重要）
  ↓
Phase 1C: 発注管理
  ↓
Phase 1D: 受注・納品・請求
  ↓
Phase 1E: AI 機能コア（メール振り分け・議事録要約・翻訳）
  ↓
Phase 2: AI 読み取り機能 + 仕様書 AI 生成
```

### 想定期間

Phase 1A 完了まで 2 ヶ月程度。
ただし、これは目安であり期限ではない。
マスターはしっかり作ることを優先する。

### AI 機能の早期投入

**やらない**。理由：

1. データ混乱のリスク
   - AI が解釈した内容を不完全なデータ構造に書き込むと、後の Phase で
     テストデータの移行・削除・整理が発生
   - 業務トランザクションの完成形が確定していない段階では、AI 判定の
     精度評価ができない
   - 中途半端なテストデータが本番 DB に紛れると、shunya 社内で実運用を
     始めた時に「これは本物？テスト？」の判別がつかなくなる

2. 「サンプル」は AI 実装時にいくらでも作れる
   - Phase 1E の時点では、Phase 1A〜1D で実運用された本物のデータが
     大量に蓄積されている
   - これは AI のテストデータとして本物のサンプルそのもの

3. 集中による品質維持
   - シナリオ A を 2 ヶ月で進めるには、マスター → 見積もり → 発注 →
     受注 → 納品請求 を集中して実装する必要がある
   - 並行で AI 機能のプロトタイプを作ると確実に期間が伸び、各機能の
     品質も落ちる

## 1A-11 → 1A-10 順序固定の理由

（2026-05-25 追記）

`DeliveryDestination.buyerId` は必須 FK（`onDelete: Cascade`）であるため、
Buyer 未実装では納品先の新規作成テストすら通せない。よって以下の順序固定：

1. Phase 1A-11（Buyer）を先に完遂
2. その後 Phase 1A-10（DeliveryDestination）に即着手

なお `Buyer.clientId` は元設計で optional（`String?`）であるため、Buyer は
Client さえ実装済みなら単独で完遂可能。

## 為替レート（1A-9）リスケの理由

（2026-05-25 追記）

当初は順次実装の対象としていたが、Phase 1A-8 完遂時点での
再評価により、以下の理由から Phase 1A 末尾（1A-13 完了後）に
リスケすることとした：

- **時系列データのマスター**： `(baseCurrency, targetCurrency, rateDate, source)`
  のユニーク制約を持つ。1 レートを「編集」するのでなく、新しい日付で
  「追加」する運用が基本。アーカイブ・復元の概念が薄い
- **CRUD パターンが shunya-master-patterns 標準と異なる**： 8 関数構成のうち
  半数（update / archive / restore / checkUsage）が意味をなさない
- **外部 API 連携が前提**： 日銀公示レート（BOJ）の自動取得を Phase 1B で
  実装する設計。手動入力 UI 単独では Phase 1B が来るまで実利用ゼロ
- **Phase 1B 見積もりエンジンとの結合度が高い**： 見積もり作成時の
  `exchangeRateAtCreation` 記録、為替変動 ±5% 検知などが Phase 1B 主要
  ロジック。為替レート + 為替モニタ + BOJ API 連携をセットで実装する方が
  価値が高い
