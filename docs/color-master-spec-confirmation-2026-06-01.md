## 0. このドキュメントの位置づけ

shunya 社内で「色」を一意に呼べる共通言語を持つための新マスター `Color` の設計確認書。本書は仮置き版であり、色名・色味（CMYK/hex）は慎太郎さんのレビューで確定させる。確定後に `model Color` を schema に追加し、seed（初期50色）と一覧/新規/詳細/編集 UI を整備する。

仕入先側のカラー表記は各社バラバラで規則性がないためマスター化しない。本マスターは自社の共通番号体系であり、仕入色は将来 Material 側で「自社番号へのマッピング」として紐付ける。

---

## 1. 背景・設計判断の記録

- 既存スキーマ全体は「色マスターなし・各テーブルが colorCode/colorName/colorHex 等を文字列で直接保持」する設計（SKU・Design・BOM・在庫ロット・受注/納品明細など）。本マスターはその中央辞書として新設する。
- 当面は Color マスターを新設し、まず Material.availableColors から参照し始める。既存インライン色列（SKU 等）の FK 化・統合は Phase 1B 以降の横断作業として切り出す。
- 色の正確な定義は CMYK を基準値として持ち、hex はそこから算出。人が呼ぶのは2桁番号。役割分担は「番号＝ユニークキー（口頭・チャット用） / CMYK＝色の正体 / hex＝画面表示・円グラフ描画」。

---

## 2. 番号体系のルール

- 2桁番号 `00`–`99`。
- **十の位＝色相系統**（10系統）。
- **一の位＝濃淡**。基本は奇数 `1 / 3 / 5 / 7 / 9` の5段（`1` が最も淡い・明るい → `9` が最も濃い・暗い）。
- **偶数 `0 / 2 / 4 / 6 / 8` はイレギュラー追加枠**として温存（例: サックスとターコイズの中間色が必要になれば `52` を差し込む）。
- 色名は業界呼称（サックス、ネイビー等）。和名（水色・紺等）は使わない。

### 十の位（色相系統）の割り当て

| 十の位 | 系統 |
|---|---|
| 0X | グレー（明・無彩） |
| 1X | レッド |
| 2X | オレンジ |
| 3X | イエロー |
| 4X | グリーン |
| 5X | ブルー |
| 6X | パープル |
| 7X | ピンク |
| 8X | ブラウン |
| 9X | グレー（暗・無彩） |

※無彩は明側 0X・暗側 9X に分割。中間グレーは 0X の濃側（09）と 9X の淡側（91）が連続するよう配置。
※水色（サックス）・青緑（ターコイズ）はブルー（5X）の淡側に統合。シアンの独立色相は設けていない。

## 4. データモデル `model Color`（Prisma 案・要レビュー）

既存マスター（MaterialCategory / ExpenseCategory 等）と同じパターン（companyId・status・timestamps・soft delete・`@@unique([companyId, code])`・`@@map`）に揃える。

```prisma
/// 自社カラーマスター（社内の色共通言語）
model Color {
  id          String   @id @default(uuid())
  companyId   String   @map("company_id")

  colorNumber String   @map("color_number") @db.VarChar(2)  // "51" 偶数はイレギュラー予約
  colorName   String   @map("color_name") @db.VarChar(100)  // 業界呼称（サックス、ネイビー…）

  hueGroup    Int      @map("hue_group")   // 十の位 0-9（色相系統）
  toneStep    Int      @map("tone_step")   // 一の位（1/3/5/7/9 基本、偶数は追加枠）

  cmyk        String   @db.VarChar(20)     // "100.85.0.40"（基準値）
  hex         String   @db.VarChar(7)      // "#001799"（算出値・表示用）

  impression  String?  @db.VarChar(100)    // 印象キーワード（任意・元画像の Dynamisch 等に相当）
  sortOrder   Int      @map("sort_order")  // 表示順（colorNumber 昇順を想定）

  status      String   @default("ACTIVE") @db.VarChar(20)
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @updatedAt @map("updated_at")
  deletedAt   DateTime? @map("deleted_at")

  @@unique([companyId, colorNumber])
  @@index([companyId, hueGroup])
  @@map("colors")
}
```

論点メモ:
- `colorNumber` は先頭ゼロ保持のため `Int` ではなく `VarChar(2)`。将来3桁拡張の可能性があれば `VarChar(3)` も検討。
- 色名の多言語（colorNameEn / Zh / Vi）は、見積 PDF・貿易書類で色名の他言語表記が要るなら追加。
  - **【確定 2026-06-29 / B-063】colorNameEn を採用（nullable・既存色は NULL で後埋め・輸出下げ札/貿易書類向け）。Zh / Vi は実需確定まで列を作らない。根拠: v0.4 §0 決定4改訂2 ＋ 本セッション方針。**
- `impression` は元のカラー円グラフ中心にあった印象語（Dynamisch / Frisch 等）に相当。意思疎通の補助として残すか要判断。

## 5. Material / 既存色フィールドとの関係

- **Material.availableColors**（現状 `Json? [{name, code, hex}]`）は、Color マスター確定後に「自社カラー番号の配列＋仕入先呼称メモ」を持つ形へ。
  - 例: `[{ colorNumber: "57", supplierColorName: "ミッドナイトブルー" }]`
  - 仕入先のバラバラな呼称を自社番号にマッピングする受け皿として機能。
- **SKU.colorCode / Design.colorPalette / 在庫ロット / 受注・納品明細**などの既存インライン色列の FK 化・統合は **Phase 1B 以降**の横断作業として別チケット化。本マスター新設では触らない。

## 6. 未確定事項（要・慎太郎さん確認）

1. **5X ブルーの並び**: サックス → ターコイズ（やや青緑）→ ロイヤルブルー（純青）と、濃淡だけでなく色相も少し動く順で確定してよいか。
2. **各色相の色名・色味**: 1X〜4X・6X〜8X の色名は仮置き。実物の生地色・実務呼称に合わせて差し替え可。
3. **無彩の分割**: 明側 0X・暗側 9X に分けた件。中間グレーの境界（09 / 91 付近）の刻みでよいか。
4. **シアン**: 独立色相を設けず青の淡側に統合（サックス/ターコイズ）。生地で水色を明確に分けたい場合は色相枠を再編する。
5. **印象キーワード `impression`**: 元画像にあった印象語を各色相に持たせるか。
6. **色名の多言語**: colorNameEn / Zh / Vi を持つか（見積・貿易書類の要否次第）。 → **【確定 2026-06-29 / B-063】colorNameEn 採用（nullable・後埋め）。Zh / Vi は実需確定まで列を作らない。**
7. **CMYK の丸め方針**: 値を `0/50/100` 等のキリの良い数字に寄せるか、実物に忠実な実値にするか。
