## 1. 業務シナリオ

```
1. shunya が工場 / 仕入先に発注（PO / WO 発行）
   ↓
2. 商品 / 素材が納品される
   ↓
3. 工場 / 仕入先から請求書（PDF / 紙）受領
   ↓
4. 発注内容と請求書の付け合わせ
   ↓
5. 整合する → 自動承認 / 投入候補として提示
   逸脱する → 経理が確認、修正、保留
```

shunya のような OEM 業務では、月数十〜数百件の請求書を処理する。手作業の効率化は経理スタッフの最重要課題のひとつ。

## 2. 「整合」の定義（4 階層）

請求書 vs 発注の付け合わせ項目は複数あり、それぞれ整合基準が違う：

| 項目 | 整合判定 | 逸脱パターン |
|---|---|---|
| **品番 / 素材コード** | 完全一致 | 別案件と混同、品番表記揺れ |
| **数量** | 完全一致 or 許容範囲内 | 減産率、追加生産、誤計上 |
| **単価** | 完全一致（PO 確定単価）| 為替変動、追加コスト |
| **合計金額** | 数量 × 単価 + 税 = 請求額 | 計算ミス、税区分相違 |

→ **「自動投入」は全 4 項目が整合した場合のみ**。1 つでも逸脱があれば人間確認が必須。

## 3. 4 つの整合状態（State 1〜4）

```
┌──────────────────────────────────────────┐
│ State 1：完全一致                        │
│ → 自動承認候補、ステージング自動投入     │
│   ただし最終登録は経理の「一括確認」必要 │
├──────────────────────────────────────────┤
│ State 2：軽微な差異（許容範囲）          │
│ → 自動承認候補として提示、経理が「OK」  │
│   許容範囲例：金額 ±1 円（端数）         │
├──────────────────────────────────────────┤
│ State 3：明確な乖離                      │
│ → 「差異あり」として保留                 │
│   経理が原因確認、修正 or 工場へ問合せ  │
├──────────────────────────────────────────┤
│ State 4：対応する PO/WO が見つからない   │
│ → 「未照合」として保留                   │
│   経理が手動で PO/WO を選ぶ              │
└──────────────────────────────────────────┘
```

## 4. 「自動投入」の意味を 2 段階に分けて整理

| ステップ | 自動化レベル | 操作主体 |
|---|---|---|
| a. AI による抽出 + 構造化 | 完全自動（AI）| システム |
| b. PO / WO との突合 | 完全自動（システム）| システム |
| c. ステージングテーブルへの保存 | 完全自動 | システム |
| d. 経理の確認 | **必須**（State 1 でも）| 人間 |
| e. 本テーブルへの確定登録 | 人間承認後に自動 | システム |

→ **「人間レビューを完全に省略する」のは推奨しない**。ただし、State 1（完全一致）の場合は経理が「一括承認」ボタンで 10〜100 件をまとめて承認できる UI にすれば、実質的に「ほぼ自動」になる。

## 5. なぜ完全自動執行を避けるか

| 理由 | 説明 |
|---|---|
| **法的要件** | 適格請求書、インボイス制度では確認責任が経理にある |
| **監査トレース** | 「誰が、いつ、承認したか」のログ必要 |
| **AI 誤抽出** | 数字の OCR ミス、桁ズレを最終チェック |
| **異常検知** | 為替急変、二重請求などのフラグ案件を見逃さない |
| **責任所在** | 「システムが勝手にやった」を許さない |

→ **State 1 でも経理が「目を通す」必要がある**。ただし UI で負荷を最小化する。

## 6. 推奨ワークフロー（Phase 1D + 1E の協調）

### システム上のデータの流れ

```
請求書 PDF 受領
   │
   ▼
[AI 抽出層]（Phase 1E）
   ├ 品番、数量、単価、金額、税区分、納品日 を JSON 化
   ├ 信頼度スコア付与
   ▼
[突合エンジン]（Phase 1D）
   ├ 品番から関連 PO/WO 候補を検索
   ├ 数量 / 単価 / 金額の差異計算
   ├ State 1〜4 のいずれかに分類
   ▼
[ステージングテーブル]
   ├ invoice_staging（仮登録）
   ├ matching_log（突合結果）
   ▼
[経理 UI]
   ├ Tab: State 1（完全一致、N 件）       ← 一括承認可能
   ├ Tab: State 2（軽微差異、N 件）       ← 個別確認、一括承認可能
   ├ Tab: State 3（差異あり、N 件）       ← 個別対応必須
   ├ Tab: State 4（未照合、N 件）         ← 手動マッチング
   ▼
[承認] → [本テーブル登録]
   ├ Invoice モデル（請求書本体）
   ├ InvoiceLine モデル（明細）
   ├ AuditLog（承認履歴）
```

### State 別の経理 UI 案

#### State 1：完全一致（一括承認）

```
┌───────────────────────────────────────────────────┐
│ 完全一致請求書（12 件）                          │
│                                  [全 12 件を承認] │
├───────────────────────────────────────────────────┤
│ ☑ #001  工場 A  ¥250,000  WO-2026-0042  ALPHA   │
│ ☑ #002  工場 A  ¥180,000  WO-2026-0043  BETA    │
│ ☑ #003  工場 B  ¥320,000  WO-2026-0044  ALPHA   │
│  ...                                              │
└───────────────────────────────────────────────────┘
```

経理は「目視で軽く確認 → 一括承認」で 1 分で完了。

#### State 2：軽微差異

```
┌─────────────────────────────────────────────────────┐
│ 軽微差異請求書（3 件）           [全 3 件を承認] │
├─────────────────────────────────────────────────────┤
│ #021  工場 C  PO 金額: ¥250,000                    │
│              請求額: ¥249,997  差異: -3 円（端数）│
│              ☑ 承認  [編集]                       │
└─────────────────────────────────────────────────────┘
```

#### State 3：差異あり

```
┌─────────────────────────────────────────────────────┐
│ 差異あり請求書（2 件）                             │
├─────────────────────────────────────────────────────┤
│ #041  工場 D  PO 数量: 500、請求数量: 480         │
│              差異: -20（減産？要確認）             │
│              [PDF 確認] [WO 確認] [編集] [保留]   │
└─────────────────────────────────────────────────────┘
```

#### State 4：未照合

```
┌─────────────────────────────────────────────────────┐
│ 未照合請求書（1 件）                               │
├─────────────────────────────────────────────────────┤
│ #061  工場 E  品番: MK-26SS-XX-099                │
│              対応 PO/WO 候補: なし                 │
│              [PO/WO を手動選択] [新規登録] [却下] │
└─────────────────────────────────────────────────────┘
```

## 7. 一括承認の安全装置

| 装置 | 動作 |
|---|---|
| **金額上限** | 1 件あたり ¥1,000,000 超は一括承認不可（個別確認必須）|
| **新規工場 / 仕入先** | 過去取引のない相手の請求書は一括承認不可 |
| **二重請求検知** | 同じ請求書番号が過去にある場合は警告 |
| **承認ログ** | 「経理 X が State 1 を 12 件一括承認、2026-06-15 14:23」を AuditLog 記録 |

## 8. Phase 別の実装範囲

| Phase | 範囲 |
|---|---|
| **Phase 1A**（マスター）| Material / ModelCode / Buyer 等のマスター実装 |
| **Phase 1B**（見積、品番、BOM）| 見積エンジン、品番、BOM 連動 |
| **Phase 1C**（発注、在庫）| PO / WO 発行（請求書突合の片側を確立）|
| **Phase 1D**（受注、納品、請求）| **Invoice 本テーブル実装、突合エンジン基本実装、手動突合 UI**|
| **Phase 1E**（AI 補助）| **PDF 自動抽出、State 分類 UI、一括承認 UI**|

→ **Phase 1D で「手動突合」、Phase 1E で「AI 自動抽出 + 候補提示」**。

## 9. 追加検討論点（Phase 1D / 1E 着手時）

### 論点 a：請求書受領経路

| 経路 | 対応方法 |
|---|---|
| メール添付 PDF | メール統合機能（Phase 1E）|
| 紙の請求書 | スキャン → PDF アップロード（手動）|
| FAX | スキャン → PDF アップロード（手動）|
| 電子請求書（PEPPOL 等）| Phase 2 以降 |

### 論点 b：複数 PO/WO に跨る請求書

```
1 つの請求書に複数案件の明細が含まれる場合
例：工場 A が 3 つの WO（WO-001, WO-002, WO-003）をまとめて 1 通の請求書で送る

→ 請求書明細レベル（InvoiceLine）で WO_ID を持つ設計が必要
```

これは元設計でカバーされているはず（仕様書 Part 2 のデータ構造）。

### 論点 c：請求書 vs 納品書の整合

```
工場 A の請求書 = 工場 A の納品書（DLV）が複数まとまっている
納品書 = PO/WO の出荷タイミングで作成

→ Invoice → DeliveryNote → PO/WO の階層
```

### 論点 d：仕入消費税 / 適格請求書

```
適格請求書発行事業者か否かで仕入控除可否が変わる
Supplier / Factory マスターの isQualifiedInvoiceIssuer フィールドで管理（既に実装済み）

突合時のチェック：
- isQualifiedInvoiceIssuer = true なら仕入控除可
- false なら経過措置適用（時限的）
```

## 10. AI 抽出の現実性（2026 年時点）

| 機能 | 実現可能性 |
|---|---|
| PDF テキスト抽出 | ✅ 完全に確立（Claude API は PDF を直接読める）|
| PDF からの構造化抽出（JSON）| ✅ 実用レベル、精度 90%+ |
| メール本文の自動分類 | ✅ 実用レベル |
| 手書き請求書の OCR | △ 精度低い、人間レビュー必須 |
| 「自動で DB に登録」（完全自動執行）| ⚠️ 技術は可能、業務的に危険 |

元設計の方針：

> **AI 補助は補助に徹する**：重要決定は必ず人間が確認する。AI は候補提示・要約・翻訳に徹し、自動執行はしない（仕様書 Part 1）

この方針は **業務的に正しい**。完全自動執行は永久にしない。
