- ただし SKU は **生成導線・一覧取得 action・描画が一切存在しない**。prisma.sku の実使用は products.ts の削除ガード .count 1箇所のみ。seed も SKU を作らない。**dev DB の skus 件数 = 0**。

## SKU 数量フィールドの素性（モデルと業務工程の対応）
- Sku の数量群（orderedQuantity 受注合計 / productionQuantity 量産発注数 / producedQuantity 生産完了 / deliveredQuantity 納品済 / defectQuantity 不良 / remainingStock 在庫）は全て **受注→量産→納品の量産ライフサイクル** の数。
- 「指定数」= orderedQuantity（受注）、「取り切り」= productionQuantity（量産発注）。両者は既に別フィールドで分離済み＝マトリクスのセル表示ロジックの話。
- **サンプルは Sku ではなく SampleProduction（SP-2026-xxxx）。** 数量マトリクスは量産側であり、サンプルは品番ページに別カードで並ぶ別系統。f24-so08 の「黒 M12/L15/XL8」も量産仕様書由来の量産数。
- 「希望数」は受注前の先方希望段階の数で **Sku に対応列が無い**（orderedQuantity は受注後の確定値）。出どころ（saagara-v2 機能 / CSV 取り込み / 先方入力）は未確定＝今は入力先を決め打ちしない。

## 決定（B-064 スコープ = C で確定）
- B-064 は **「色×サイズ × (受注/量産)数 の表示 ＋ 空状態 UI」のみ**。スキーマ変更ゼロ・書き込み action なし・最低リスクで先頭に置く（v0.2 決定3 の視覚アンカーの狙いを保つ）。
- 不採用: (A) B-064 に SKU 登録 CRUD を畳む＝先頭最低リスクの狙いを壊す。出どころ未定のまま入力先を Sku 決め打ちすると後で取り込み連携に切替時に作り直し。 (B) SKU 生成を先行タスク化＝視覚アンカーが後ろにずれる。
- 希望数の入力・SKU 供給導線（saagara-v2 / CSV / 先方入力）は **出どころ確定後に別タスク**。手入力が要る場合も B-064 本体には畳まず小タスクで足す。
- ⚠️ 留保: 本決定は v0.1 §1「数量マトリクス＝Sku」対応＋実スキーマからの推論。v0.1 本文未読のため、v0.1 §1 が「数量はカルテで入力」と明記していれば A 寄りに再検討の余地。次に v0.1 本文を読める時に突き合わせる。
