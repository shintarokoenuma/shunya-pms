## 1. 確定事項 H1〜H4

### H1. PDF 生成方式(✓ 確定)
- @react-pdf/renderer(サーバーサイド生成)+ Noto Sans JP(OFL・サブセットを assets 同梱)。
- MVP 計画書の採用スタックどおり。Puppeteer(Chromium)は Railway 運用コストが高く不採用。

### H2. 生成・保存の段階化(✓ 確定)
- S-4c-2: オンデマンド生成・ダウンロードのみ。「生成(PDFドキュメント組み立て)」と
  「出力先(レスポンス or ストレージ)」をコード上分離し、後からストレージを差し込める構造にする。
- B-053(運営開始前・必須): Google Cloud Storage 統合。生成 PDF をバケット保存し
  woPdfUrl / poPdfUrl / woPdfGeneratedAt 等に記録。GCS バケット作成・サービスアカウント発行は
  慎太郎さんの GCP コンソール操作(着手時に Claude が手順書を用意)。
- 保存をレイアウト確定後に回す理由: H3 の修正サイクル中に旧版 PDF がストレージに溜まるのを避ける。

### H3. 帳票レイアウト(✓ v1.0 最小・実物レビュー前提)
- A4 縦・1伝票1枚(明細多数時は複数頁可)。
- ヘッダ: 表題「発注書」/ PO・WO 番号 / 発注日 / 発注先(御中)/ 発注元(shunya 社判ブロック:
  社名・住所・TEL/FAX・MAIL)。
- 明細表: 品名 / 品番(supplierItemCode 等) / C#(カラー) / 数量・単位 / 単価 / 小計
  — S-4c-1.6 内訳と同一語彙。金額未定明細は「未定」表記。
- フッタ: 合計(金額未定は除外・注記)/ 希望納期 / 備考(description)。
- 発注元情報の取得元: Company テーブルに住所等が無ければ当面 src/lib/constants/company-profile.ts
  に定数で持つ(設定画面化は B 起票)。実装時に Company schema を grep して判断・報告。
- ★v1.0 は最小で出し、dev で実物 PDF を見ながら v1.1 追補で調整する(慎太郎さん意向)。

### H4. スコープ外(✓ 確定)
- 送付フロー(email/fax)= B-049(既存)。
- 多言語出力(outputLanguage)= 当面 JA のみ。
- 縫製仕様書そのものの帳票出力 = B-054(新規)。最終フェーズで本腰を入れて詰める(慎太郎さん意向)。
  参照原本の Excel 構造(付属表/サイズ表/詳細図タブ)はその際の設計材料。

## 2. 本書で起票したバックログ
- B-053: 発注書 PDF の GCS 保存統合(運営開始前・必須)。バケット/サービスアカウント/Railway 環境変数。
- B-054: 縫製仕様書の帳票出力。現行 Excel(付属表・サイズ表・詳細図)の構造を原本として設計。
  B-039(規格構造化)/見積もりエンジンと関連。最終フェーズ。
