# サンプル見積 構想メモ v0.1 (2026-06-30)

- 種別: 構想メモ（spec 確定版ではない・設計論点の固定が目的）
- 位置づけ: QE 系のうち「サンプル見積」の入口構想。量産見積（QE-1 v1.0・取り切り）とは別物。
- 作成経緯: 2026-06-30 セッション。当初 QE-1 を量産前提で v1.0 push 後、対象がサンプル見積だったと判明し方向転換（下記「対象の確定」参照）。
- 上位/関連: qe-0 v1.2 / id-map-audit v0.1 要件A / product-sample §6-2（初期費用・売り立て区分）/ B-038（AI 費目マッピング）/ B-046（自然言語発注）

## 0. 対象の確定（最重要・取り違え防止）
- **量産見積（QE-1 v1.0）** = 取り切り計算・原反単位・productId 起点。qe-0 §3 で「取り切りは量産見積専用」と定義済み。**今回の対象ではない**（v1.0 は温存・将来タスク）。
- **サンプル見積（本メモの対象）** = 仕様・素材確定前後に客へ出す金額。指定数・メーター発注ベース。量産とは source・計算・スコープが別物。

## 1. なぜ見積もりが必要か（目的）
細かい仕様や素材が決まる前に、ざっくりしたコストを出して**交渉のテーブルに乗せる**ため。
→ 時間軸の先頭は (あ)。(あ) が無いと商談が始まらない。

## 2. サンプル見積は2段階
| 段階 | 何か | タイミング | source | 金額の性質 |
|---|---|---|---|---|
| **(あ) 概算見積** | ざっくり見積もり | 仕様・素材**確定前** | 経験値・過去実績（BOM 連動計算はしない） | 概算 |
| **(い) 最終見積** | 最終見積＝**納品金額** | 仕様**確定後**・納品時 | 原価（PO/WO＋BOM）＋利益/管理費 | 確定（原価＋利益＝(い-b)） |

- (あ) で交渉 → 仕様確定 → (い) で納品金額を確定、という一本の流れ。
- (い) は実質「見積もり」でなく納品金額（請求額）。

## 3. (あ) 概算見積に含めたい能力（今日リッチ化・要設計）
当初「手入力の器＋過去引き写し」と整理したが、慎太郎さんの要望でより広い:
1. **過去調達実績の検索・引き当て** — 過去に仕入れた素材コスト・生地・工賃の実額を検索し、今回の概算へ引き当てる。「この生地は前回いくら」を呼び出す。
2. **品番からの候補提案** — 品番（カテゴリ・アイテム種別）が決まれば必要な付属・工賃が大体決まる→システムが候補提示。「このアイテムならファスナー/ボタン/縫製工賃はこのあたり」。
- ★1・2 は B-038（見積→マスター化→候補提示）・B-046（自然言語発注）と地続き。(あ) 単独で閉じず、提案エンジンの入口になる。
- 手入力(あ-i)＋過去引き写し(あ-ii)は確定。粗 BOM 計算(あ-iii)は重いため (あ) では不採用＝(い) 側に寄せる。

## 4. 初期費用（絶対防衛線・確定）
- 初期費用 = マスターパターン代・プリント版代・パンチ代・サンプル代。サンプル時請求がほとんど（量産時の版代/パンチ代請求は稀）。
- 製品単価にインクルーズしない別途請求＝**1枚原価に合算しない**。別枠「別途請求項目（1枚原価外）」に項目別表示。
- 構造的根拠: `billingClassification = INDIVIDUAL_BILLING`（個別売り立て＝パターン代/版代/型代/刺繍パンチ代/グレーディング代）で分類済み（product-sample §6-2 / s-4a）。1枚原価の分子は「製品単価インクルード費目のみ÷枚数」。
- 慎太郎さんの線引き: ①加工費(WO)②材料費(PO/BOM)が多少混ざるのは許容／**③初期費用が1枚原価に混ざるのは絶対ダメ**。
- 版代・パンチ代の PO は `PoItem.isPhysicalAsset`（現物資産フラグ・実在）で識別可能。

## 5. 混在通貨・カット代（既出・関連確定事項）
- 混在通貨: 行ごとに JPY/USD（ベトナム例＝生地JPY・工賃/プリント/検品/刺繍USD 等）。BomItem/PoItem/WoItem に currency 列実在。基本通貨はドル円のみ・CNY/VND/EUR は換算 v1 対象外。
- カット代: 指定数（メーター買い）は生地代に加えカット代（裁断手数料）が乗る。専用フィールドは作らず**明細行（費目=加工）で表現**が確定済み（s-4b-1 §3）。取り切り（反単位）はカット代なし。

## 6. 次セッションの入口（優先順）
1. (あ) の2能力（過去実績の検索・引き当て / 品番からの候補提案）のどちらから・どの source で作るかを詰める。B-038/B-046 との関係を整理。
2. 検索引き当ての引き元（過去のサンプル見積/納品額(い)が保存される実体がスキーマにあるか）を recon。無ければ「(い)を保存する箱」の依存が出る。
3. (あ) の永続化要否（保存して(い)に引き継ぐか・出力して終わりか）。
4. (い) 最終見積＝納品金額（原価＋利益(い-b)）のマージン層の持ち方。Quotation 系スキーマのマージン列・元設計 quotation_engine の「マージン4階層」を recon してから設計。

## 7. 本日の学び（再発防止）
- 対象スコープの取り違え: 引き継ぎメモ要約「QE-1＝量産見積」を鵜呑みにし、量産・取り切り前提で v1.0 を push・接続点 recon 2往復してから、qe-0 §3「取り切りは量産専用／サンプルは指定数」の排他文に気づいた。spec 原文は recon で手元にあったのに要約フレームで読んだのが原因。
- 対策: design-reread に Step 0「対象確定ゲート」を追加（着手前に対象を spec 原文で確定・排他文を grep・食い違えば本文前に1問確認）＋「重い成果物の前に対象を1問で確定」規律を追加。本日スキル更新済み。
- 因果の取り違え: (あ)(い) の関係を「(い)が一次・(あ)は参照」と組んだが逆。(あ)＝交渉前の概算が時間軸の先頭・目的そのもの。「source をデータに紐づけよう」とし続けたのが (あ) の本質（データ確定前だから概算）と衝突していた。
