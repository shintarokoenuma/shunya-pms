## 0. 経緯（なぜ v1.0 を作り替えるか）

PR #64（骨格版 S-4b-1）をローカルで実際に操作した結果、明細フォームが「サンプル発注の実務」に追いついていないことが判明した。具体的には:

1. 生地・付属の明細には、仕入先品番・カラー番号・デザイン番号・サイズ等の詳細項目が必要。
2. 単位が多様（個・m・反・一式 …）。
3. **金額未定のまま発注することが多い**（見積もりは後から届く）。現状の `unitPrice`/`subtotal` NOT NULL では発注できない。
4. 発注書は**発注先単位で1枚**にまとめて出す運用（現状は仕様書PDFを読み取って一括発注）。

これらを反映し、明細を「サンプル発注実務カード」に作り替える。本書 v1.1 がその確定仕様。

## 1. 確定事項（F1〜F8）

### F1. 発注書の単位（✓ 確定）

**1 PO = 1 発注先 = 発注書1枚。** 同一発注先の複数資材は1 POの複数明細にまとめる（B社のボタン＋ファスナー）。これにより「発注先ごとにまとめて発注書を出す」現運用を shunya 側から正規に再現できる。schema 構造（1 PurchaseOrder → 複数 PoItem）はそのまま。

### F2. 起点UIと紐付け（✓ 確定・v1.0 踏襲）

タスク行起点（progressTaskId 紐付け）を維持。PO 対象は FABRIC/TRIM/BODY。今回の発注例は各 PO が単一の資材種別に収まる（B社の2明細はどちらも TRIM）ため、タスク紐付けを壊さず実務を吸収できる。
- 「SP に資材を全部入れて発注先別に自動で PO 分割」のような発展形は将来構想（B-038 系）。

### F3. サイズ と 規格 の言葉の定義（✓ 確定）

慎太郎さんの定義を本仕様の用語とする:
- **サイズ** = こちらが指定できるもの（ファスナー長・ボタン径・製品の企画サイズ展開）。
- **規格** = こちらが変更できないもの（生地幅・巻きメーター数・エレメント素材など、相手/物の属性）。規格は主に**用尺計算と輸出**で使う。

### F4. 明細（PoItem）の確定項目（✓ 確定）

| 項目 | 区分 | 持たせる場所 | カラム | migration |
|---|---|---|---|---|
| 品名 | — | 素材マスター select **or** 自由入力 | materialId / customItemName（既存） | — |
| 仕入先品番（20000・B-100・FC-DHL） | コード | PoItem | **`supplierItemCode`（新規）** | ADD COLUMN |
| デザイン番号（D/#A・DHL） | コード | PoItem | **`designCode`（新規）** | ADD COLUMN |
| カラー番号（C/#100。テープカラー等もここ） | 色 | PoItem | `colorCode`（既存・流用） | — |
| **サイズ**（ファスナー長・ボタン径） | サイズ | PoItem | **`sizeSpec`（新規・文字列）** | ADD COLUMN |
| 仕様・規格の自由記述 | 規格/仕様 | PoItem | `specification`（既存・流用） | — |
| 数量 | — | PoItem | quantity（既存） | — |
| 単位 | — | PoItem | unit（既存・**UIは候補プルダウン＋自由入力**） | — |
| 単価 | — | PoItem | unitPrice（**nullable 化**） | DROP NOT NULL |
| 小計 | — | PoItem | subtotal（**nullable 化**・単価未定時 null） | DROP NOT NULL |
| 費目 | — | PoItem | costCategoryId（S-4a） | — |
| 売り立て区分 | — | PoItem | billingClassification（S-4a） | — |
| 現物資産 + 保管日付 | — | PoItem | isPhysicalAsset / assetStorage*（S-4a） | — |
| メモ | — | PoItem | notes（既存・流用） | — |

- **製品の企画サイズ展開（S/M/L・1/2/3）は明細に持たせない**。これは Product 側の属性（SKU・数量モデル＝B-030 と地続き）。本書スコープ外（B-041 へ）。
- **生地幅・巻きメーター数（規格）は当面 `specification` に文章で集約**。数値構造化は B-039。

### F5. サイズ（`sizeSpec`）の持ち方（✓ 確定）

- **数値分解せず文字列1本 `sizeSpec`（VarChar100）** で持つ（例 `20cm` / `15mm` / `20L`）。
- 理由: 単位混在（cm/mm/リーニュL）があり、数値＋単位に割っても資材種別で絞らないと統計が成立せず旨味が薄い。統計・候補提示の主軸は**仕入先品番・素材・色**（既に構造化済み）で、サイズはその資材を特定したうえの付随属性。
- デメリット（範囲検索・集計が不可）の緩和策として、UI に**表記サジェスト**（`cm`/`mm`/`L` のプレフィル例）を付けて表記ゆれを抑える。
- 数値化（`sizeValue`+`sizeUnit`）は、データが貯まってから後付けで設計・バックフィル可能。**B-039 に予約**。

### F6. ファスナー等の複雑資材の扱い（✓ 確定）

ファスナーは「スライダー（サイズあり）/ エレメント（カラー・素材・サイズを内包）/ テープカラー / 長さ（オーダー可）/ 仕様（オープン・止め・逆開）」と多軸。**専用構造化はしない**（ファスナー専用カラムを増やすと他資材で全 null になり共通テーブルを歪める）。
- 色（テープカラー）→ `colorCode`。
- 長さ・ボタン径など指定値 → `sizeSpec`。
- スライダー / エレメント素材 / 仕様（オープン止め等） → `specification` に自由記述（例「エレメント金属/スライダーDA8/オープン」）。
- 軸ごとの構造化（ファスナー分解）は **B-040** に予約。

### F7. 金額未定対応（✓ 確定）

`unitPrice` / `subtotal` を **nullable 化**。null＝未定。単価が入れば `subtotal = quantity × unitPrice` を計算保存、未入力なら両方 null で保存し画面は「未定」表示。見積もり到着後に単価を入れて更新する運用。
- 既存行は値を持つため DROP NOT NULL は安全（本番 `po_items`=0・dev も検証掃除済みで0）。

### F8. 発注書PDF（✓ 確定：今回は項目を構造で持つまで）

ご指摘どおり PDF 出力は明細構造に直結する。**「1 PO=発注書1枚」を確定し、PoItem に必要項目（発注先・社内品番・仕入先品番・色・サイズ・仕様・数量・単位・単価・納期）を構造で漏れなく持つ**ことが本書の範囲。PDF 化の実装自体は後段（S-4c か専用段）。これが「仕様書PDFを読み取って一括発注」を shunya 側から正規出力する形へ移行する土台になる。

## 2. migration（すべて非破壊・要順守）

```sql
-- 1. 仕入先品番（任意）
ADD COLUMN supplier_item_code VARCHAR(100) NULL;
-- 2. デザイン番号（任意）
ADD COLUMN design_code        VARCHAR(100) NULL;
-- 3. サイズ（指定値・文字列。ファスナー長/ボタン径 等。任意）
ADD COLUMN size_spec          VARCHAR(100) NULL;
-- 4. 単価 未定可
ALTER COLUMN unit_price DROP NOT NULL;
-- 5. 小計 未定可
ALTER COLUMN subtotal   DROP NOT NULL;
```

- Prisma schema 側: `supplierItemCode String? @map("supplier_item_code") @db.VarChar(100)` / `designCode String? @map("design_code") @db.VarChar(100)` / `sizeSpec String? @map("size_spec") @db.VarChar(100)` を PoItem に追加。`unitPrice` / `subtotal` を `Decimal?`（nullable）に変更。
- **非破壊の根拠**: ADD COLUMN（nullable）= 既存行に影響なし。DROP NOT NULL = 制約を緩める方向で既存行に影響なし。DROP / 型変更 / 既存 NOT NULL 化 は含まない。
- **段階停止プロトコル（S-4a と同一）**: dev に `migrate dev` で適用・空 diff 確認 → **本番投入前に Claude.ai 側で一度止める** → 承認後に本番（マージ＝`migrate deploy`）。

## 3. カット代等の付帯費用（✓ 確定）

専用フィールドは作らず、**明細行で表現**する。例（A社の生地）:

```
明細1: 表地    数量3  単位M   単価(未定)  費目=本体生地
明細2: カット代 数量1  単位一式 単価=確定額  費目=加工 等
```

生地本体が未定価格でもカット代だけ確定で出せる、という実務に合う。発注書にもそのまま行として出る。

## 5. 後続バックログ（本追補で起票）

- **B-038**: 見積もり → 量産マスター化 → 候補提示。サンプル時に届いた見積もりの単価を、素材マスターの標準単価候補として自動で育てる構想（AI 費目マッピングと地続き）。今回 PoItem に費目・素材・単価・詳細を構造化して持たせることが基盤。
- **B-039**: 規格・サイズの構造化（数値統計・用尺計算・輸出）。`sizeSpec` の数値化（`sizeValue`+`sizeUnit`）、生地幅・巻き/反の数値化はここ。データが貯まってから後付け・バックフィル。
- **B-040**: ファスナーの軸分解（スライダー/エレメント/テープ/長さ/仕様）。発注頻度と要件が固まってから。
- **B-041**: 製品サイズ展開（S/M/L）を Product 側で扱う（SKU・数量モデル＝B-030 と統合検討）。
