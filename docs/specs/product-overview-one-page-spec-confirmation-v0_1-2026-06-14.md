## 0. 北極星
品番カルテ詳細ページ（src/app/(app)/products/[id]/page.tsx）の上に、f24-so08「量産用」シート相当の概要を1ページで再現する。1ページで品番の概要が掴める状態を目指す。

## 1. 構成5要素と現状
1. 品番（品名・ブランド・パターンNo・シーズン）= Product。既存。
2. 数量マトリクス（カラー×サイズ：黒 M12/L15/XL8/計35、ベージュ 6/7/4/17）= Sku（色×サイズ＋数量群）。データは既存・ページ未描画。
3. カラー展開（A=ブラック/B=ベージュ＋色名）= 現状 Sku の色（文字列・マスター未連携）。色名はマスター依存（後述 §3）。
4. 付属マトリクス（表地/芯地/リブ/フロントZIP/ハトメ/パッキン/スピンドル × カラーウェイ）= β。現 BomItem は色1組・Product 直結でカラーウェイ軸なし。最大の穴。
5. 絵型（縫製図・サンプル写真）= 画像アップロード基盤（B-027）。未着手。

→ 器（縦積みカードのページ骨格）は既存。欠落は「数量マトリクス表示・付属マトリクス(β)・色名供給・絵型」の4点。

## 2. 付属マトリクスの列構成（本セッション確定）
列: 使用箇所 / 仕入先品番 / サイズ・用尺 / 先方カラー品番(C/#)×カラーウェイ。
- 左3列（仕入先品番=BomItem.supplierItemCode、サイズ=sizeValue/sizeUnit、用尺=usagePerUnit/unit）は色共通のベース。
- 先方カラー品番(C/#)のみカラーウェイ別 → β のデータ構造（色非依存ベース＋カラーウェイ×色マトリクス）を裏づけ。
- 実例(f24-so08): 表地 08500026013 用尺1.376m → 黒 C/#099 / ベージュ C/#200。リブ 27-Y9100 0.57m → C/#56 / C/#14。品番は色共通・C/# のみ変動。

## 3. 色の二層モデル（color-master 仕様確認書 2026-06-01 を読み直して確定）
- 仕入色（先方カラー番号 C/#）はマスター化しない（各社バラバラ・規則性なし）。調達カラーとして BOM/PO が文字列保持。発注書に書く値。
- Color マスター（自社2桁番号）＝認識・色名の権威源泉。サンプル指示書・量産仕様書・工場指示書・下げ札の4帳票すべてで色名はマスター必須（番号のみの先方カラーから人が読める色名を出せる唯一の源泉）。
- 両者は別レイヤー。橋渡しは発注のたびではなく素材ごとに一度（Material.availableColors の写像）。
- 帳票の色名解決点 = 製品カラーウェイ ⇔ マスター色 の必須紐付け。カラーウェイの実体は現状 Sku の色次元。

### 3-1. ドキュメント改訂提案（2点・要確定）
- (改訂1) Material.availableColors を「仕入先名」写像→「仕入先番号」写像へ。例 [{ colorNumber:"57"(自社), supplierColorCode:"099"(先方番号) }]（先方は番号のみのため）。
- (改訂2) Color マスターに colorNameEn 追加（下げ札・輸出向けに英語色名が必要なため。§6-6 の「未定」を「要る」側に確定）。

## 4. C（BOM⇔製作セット紐付け）= 不要確定（取り下げ）
理由（3方向から）: ①運用上の回次差は実質「色違い」でカラーウェイ軸 ②理想ページに出る軸はカラーウェイであって SP でない ③ページ構造上もサンプル製作セットカードと BOM カードは別系統で縦並びするだけ。よって C は実装しない。

## 5. 構造論点（未決）: BOM 2系統
